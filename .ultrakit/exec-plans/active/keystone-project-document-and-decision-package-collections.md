# Keystone Project Document And Decision-Package Collections

## Purpose / Big Picture

Keystone already exposes the project collection routes in the public `v1` contract, but two of them still return empty typed stubs:

- `GET /v1/projects/:projectId/documents`
- `GET /v1/projects/:projectId/decision-packages`

After this plan lands, those routes will return real project-scoped collections instead of empty placeholders. Operators and the eventual UI will be able to inspect artifact-backed decision-package history for a project and browse document-like project notes generated by actual runs, while the rest of the project and run APIs stay stable. To make those reads honest and queryable, artifact refs will also start carrying a real `projectId`.

The observable outcome is:

- project decision-package collections are real and no longer empty scaffolds,
- project document collections return conservative, document-like entries derived from durable run artifacts instead of an always-empty stub,
- `GET /v1/decision-packages/:decisionPackageId` is upgraded from a stub to the same real projection model so project decision-package rows have a consistent detail surface, and
- README, developer docs, and API tests stop describing these routes as backend gaps.

This plan is intentionally narrow. It does not introduce a full project-document CMS, a first-class decision-package authoring workflow, or broader run-launch changes.

## Backward Compatibility

Backward compatibility is **required** for this work.

The user asked to implement the missing project APIs, not to redesign the surrounding contract. The plan must therefore preserve:

- existing route paths,
- existing `data` / `meta` envelope shapes,
- current project CRUD behavior,
- current `POST /v1/runs` behavior, including inline decision-package launch as the only supported write path,
- current run, artifact, and approval APIs.

The allowed behavior change is inside the two currently empty project collection routes and the matching decision-package detail route:

- list responses can change from empty stub collections to populated projected collections,
- scaffold metadata can change from `stub` to `projected`,
- README and developer docs should stop calling these exact routes “missing” once the new behavior is landed,
- artifact-ref persistence can grow a new `project_id` field as an internal schema change, provided the public API contract remains stable.

## Design Decisions

1. **Date:** 2026-04-18  
   **Decision:** Implement the missing project collection routes as read-only projections over already durable run/session/artifact state instead of introducing new public write APIs.  
   **Rationale:** The user explicitly asked for the missing project APIs only. The current system already persists enough run metadata and artifact history to back a useful first cut without reopening project configuration or run-launch authoring flows.  
   **Alternatives considered:** Adding new public document and decision-package create/update/delete endpoints; leaving the routes as empty stubs.

2. **Date:** 2026-04-18  
   **Decision:** Add `project_id` to `artifact_refs` and require all artifact creation paths to persist it.  
   **Rationale:** The user explicitly wants artifact refs to carry project identity. That makes the new project APIs direct and durable instead of depending on indirect `project -> run -> artifact` reconstruction from run-session metadata. It also gives future project-scoped artifact queries a clean primary key path.  
   **Alternatives considered:** Keeping project scoping indirect through run sessions only; storing `projectId` only inside artifact metadata JSON.

3. **Date:** 2026-04-18  
   **Decision:** Project decision-package collections and decision-package detail will be sourced from persisted `decision_package` artifacts associated with runs belonging to the target project.  
   **Rationale:** `RunWorkflow` already writes durable `decision_package` artifacts for every compile path, and this plan now makes project identity explicit on each artifact ref. That makes decision-package history a natural projection target without a separate decision-package catalog.  
   **Alternatives considered:** Adding a first-class `project_decision_packages` table; deriving decision packages only from run-session metadata without reading artifact bodies.

4. **Date:** 2026-04-18  
   **Decision:** Project documents will be an artifact-backed projection of only conservative, document-like artifact kinds: `task_handoff` -> `task_contract`, `run_note` -> `review_note`, and `run_summary` -> `other`.  
   **Rationale:** The UI-first contract already has a `ProjectDocument` resource, but there is no honest dedicated project-document store yet. Using only clearly document-like artifacts avoids falsely labeling execution artifacts as product specs or ADRs while still replacing the empty stub with durable project-scoped notes.  
   **Alternatives considered:** Adding a first-class `project_documents` table now; continuing to return an empty collection; aggressively mapping more artifact kinds such as `task_log` or `run_plan` into document categories they do not cleanly represent.

5. **Date:** 2026-04-18  
   **Decision:** Decision-package collections will deduplicate by `decisionPackageId`, keep the newest successfully materialized artifact per id, and order results newest-first by artifact creation time.  
   **Rationale:** The same decision package may appear across multiple runs. The collection should behave like a current project history/catalog rather than a raw append-only artifact dump.  
   **Alternatives considered:** Returning every artifact instance; deduplicating by run id; ordering lexicographically by id.

6. **Date:** 2026-04-18  
   **Decision:** Artifact-backed project projections fail soft on unreadable artifact bodies during list operations: skip unreadable list items, but return `404` from decision-package detail when no valid matching artifact can be materialized.  
   **Rationale:** One missing or malformed artifact body should not make an entire project collection unusable. Detail reads, however, should stay strict when the caller asks for one specific decision package id.  
   **Alternatives considered:** Failing the entire collection with `500`; silently inventing partial placeholder data when artifact content cannot be parsed.

7. **Date:** 2026-04-18  
   **Decision:** `POST /v1/runs` will stay inline-only in this plan even after project decision-package collections become real.  
   **Rationale:** The user asked for the missing project API reads. Expanding the run launcher to resolve `decisionPackage.source = "project_collection"` would turn this into a broader run-intake feature plan. The current inline path remains valid and backward-compatible.  
   **Alternatives considered:** Enabling `project_collection` launch in the same plan; storing a new decision-package catalog with write-through run creation.

## Surprises & Discoveries

- The “missing” project APIs are not absent routes; they are mounted routes that currently return empty typed stub collections from `src/http/api/v1/projects/handlers.ts`.
- Decision-package history is already durable enough to project because compile persists a real `decision_package` artifact for each run before task fanout.
- Project-document semantics are less direct than decision packages. There is no dedicated project-document store today, so the honest first cut is still a conservative projection over document-like artifacts and notes.
- The user explicitly wants `projectId` on artifact refs. That invalidates the earlier no-schema-change assumption and makes a small artifact-ref migration part of the real scope.
- The route contract already distinguishes `reused`, `projected`, and `stub` resources. This plan should upgrade the exact missing surfaces from `stub` to `projected` instead of inventing a fourth classification.
- `rtk npm run build` still hits the known host-bound Wrangler/Docker write boundary inside the sandbox on this machine, but the exact same build passes when rerun outside the sandbox.

## Outcomes & Retrospective

## Plan of Work

The work proceeds in three steps.

First, add the artifact-ref persistence foundation the new APIs actually need. `artifact_refs` gets a real `project_id`, the migration is applied, and every artifact creation path starts writing that field. In the same phase, add the project-scoped lookup and projection helpers the handlers will use. The backend already knows how to project runs, tasks, and workflow graphs from durable state; the missing project APIs need the same pattern. This phase adds the project-level artifact lookup helpers and the conservative mapping rules that translate decision-package artifacts and document-like artifacts into the existing `v1` resource schemas.

Second, wire the actual API reads. The project collection handlers stop returning empty collections and instead use the new projections. The decision-package detail handler also moves to the same projected source so collection items and detail reads agree about what a real decision package is in this first cut. Route and contract tests then move from “empty stub” assertions to “artifact-backed projected data” assertions.

Third, close the loop in docs and placeholder messaging. README, developer docs, the documentation placeholder copy, and the explicit technical-debt entry all need to stop describing these routes as missing while still being honest that project documents are only a conservative artifact-backed projection rather than a full living-doc workflow.
